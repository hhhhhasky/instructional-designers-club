## 〇、网站定位（一切优先级的出发点，务必先读）

> ⚠️ 本网站的定位与普通"对外官网"不同，**所有需求的取舍都以这里为准**。

1. **唯一使命：老用户维护运营**。网站存在的全部价值，是让**已付费的老用户更多地观看课程、持续学习**。这是评估每个需求价值的最高准绳。
2. **不做拉新、不做转化、不做下单**。新用户的获取、转化、订购、付款**全部在外部内容平台**（公众号、视频号、小红书、抖音等）完成。本站**不承担**任何获客与转化职能。
3. **不做 GEO/SEO**。无需为搜索引擎优化投入（结构化数据、内容预渲染、OG 图等均非目标）。
4. **运营者 = 用户本人 = 非技术人员，且是唯一运营者**。任何"需要改代码才能运营"的设计都是反目标。**一个强大、易用、覆盖面广的运营后台**是网站能否独立运转的前提。
5. 因此，主页（用户每次回来的第一站）的核心任务只有两件事：
   - **让老用户 3 秒内回到课程**（降低"继续学习"的摩擦）；
   - **给老用户持续回来的理由**（上新、直播、活动、学习成就等回访钩子）。

> 基于以上，v1.0 中"对外转化/SEO"方向的需求已**退役**（见第五节退役区），重心转向**学习留存**与**运营后台**两条主线。

## 三、问题诊断（按"老用户维护"视角重排）

### A. 看课摩擦大（核心痛点）
1. **老用户回来，首页无"继续学习"入口**：要手动进 Header 下拉 → 我的学习，或翻屏到课程区。每次回来都重复"找课"动作，摩擦直接劝退观看。
2. **课程展示只取前 4 门**：会员课程体系庞大（Plus 70+ 节、Pro 30+ 节），但首页只露 4 门，老用户难以"发现自己没学过的"。
3. **无搜索 / 无"我的收藏 / 稍后看"**：课程多但检索能力弱。

### B. 缺乏回访理由（不来看 = 不看课）
4. **无"上新 / 动态"入口**：课程更新、直播预告、活动信息散落，首页没有任何"俱乐部在更新"的信号，老用户没有回访动力。
5. **无学习激励可视化**：连续学习、本周时长、学分、成就——这些"勾人回来"的留存经典手段，前台几乎不体现（排行榜仅藏在管理后台）。
6. **无运营触达通道**：运营者无法主动告诉老用户"新课上线""本周直播"。

### C. 运营无法独立运转（反目标）
7. **首页内容几乎全部硬编码**：俱乐部介绍、创始人数据、会员简介、FAQ、评价均写死在 `HomePage.tsx`，运营改一个字都要改代码发版。
8. **资源中心 40+ 篇文章硬编码**（`ResourcesPage.tsx`），同样无法自助维护。
9. **运营者是非技术人员且是唯一运营者**：没有强大后台，网站运营就完全停滞——这是当前最大的反目标。
> 已具备：课程创建/编辑、视频 URL 录入、用户等级管理（见 2.2），无需重复。

### D. 内容质量与一致性
10. **会员人数硬编码 500**（`db/api.ts:411`），与 Hero「几百位」、Plus「400+」口径冲突。对内展示给老用户也要真实、一致。
11. **创始人数据含糊且硬编码**（`HomePage.tsx:285-289`：数百节/数千名/数万名）。
12. **「运行天数」客户端实时计算**（`HomePage.tsx:623`），首屏跳变。
13. **会员评价为 5 张图片**（`HomePage.tsx:714`），不可检索、加载慢。

### E. 工程质量
14. **`HomePage.tsx` 800 行单文件**，难维护。
15. **`getClubStats` 无缓存**（学习数据有 5 分钟 TTL，统计没有）。
16. **课程点击/访问控制逻辑重复**（主页 vs 课程中心，未抽 hook）。

### 🟥 P0 · 必须做（直接决定核心使命：让老用户看课 + 运营能自助）

#### R-P0-01　老用户个性化"学习首屏"（最高 ROI 留存设计）　【阶段一首项】
- **背景**：网站唯一目标是让老用户看课，但主页对所有访客展示相同的对外落地页，老用户回来要翻过一堆介绍才能到课程。`getLearningData()` 已具备"最近学习/系列进度"数据，但只用在 `/learning` 页，主页完全没用。
- **目标**：已登录老用户进入首页，**首屏即"继续上次学习 + 学习进度 + 推荐下一步"**，3 秒内点进课程。
- **方案概要**（依据决策 A）：
  1. HomePage 顶部、`user` 存在时插入 `<MemberHomeHero>` 个性化区：
     - **继续学习**：最近 1–3 门（带进度条 + 「继续」按钮，直跳课程详情）；
     - **学习概览**：总学分 / 已完课 / 进行中（复用 `getLearningData().overview`，仅自己可见）；
     - **推荐下一步**：基于 `seriesProgress`，推荐当前系列下一门未学课程；
     - **最新动态**：上新/直播（复用 R-P0-03 数据）。
  2. 现有介绍型区块**整体下移保留**（服务新付费用户首次到访 + 老用户身份认同），不删除、不重定向到 `/learning`。
  3. `user` 不存在时维持现有 Hero。
- **涉及文件**：`src/pages/HomePage.tsx`、新增 `src/components/home/MemberHomeHero.tsx`、`src/db/api.ts`
- **验收标准**：① 登录后首屏可见"继续学习"且一键跳转；② 进度/概览数据与 `/learning` 一致；③ 未登录视图不变。
- **工作量**：L　**价值**：★★★★★　**状态**：已实现（待手测上线，2026-06-13）

#### R-P0-02　强大的内容运营后台（运营独立运转的前提）
- **背景**：运营者=用户本人=非技术人员且是唯一运营者，但首页绝大多数内容硬编码，运营无法独立开展。这是当前最大反目标。
- **目标**：运营在后台用**表单**即可维护**全部可变内容**，全程不碰代码、不发版。
- **方案概要**（依据决策 B，已扣除既有能力）：
  1. Supabase 建内容表：`site_content`（键值化首页文案/创始人数据/价值观）、`member_profiles`（会员风采）、`faqs`、`testimonials`（会员评价）、`announcements`（动态/上新/公告，与 R-P0-03 共用）、`activities`（活动/直播）、`resources`（资源中心文章）。
  2. `AdminPage` 新增「首页内容 / 会员风采 / FAQ / 评价 / 动态 / 活动 / 资源文章」管理 section（复用 `CourseManagementSection` 表单模式与 `admin-api`）。
  3. **易用性强调**：可视化表单、必填校验、所见即所得预览、软删除+确认（防误删，单人运营风险高）；富文本用于长文案。
  4. **已有、不重复建设**：课程创建/编辑（`CourseManagementSection`，含视频 URL）、视频 R2 上传（脚本 + 后台填 URL）、用户会员等级管理。
- **涉及文件**：`supabase/`（迁移与 RLS）、`src/db/api.ts`、`src/db/admin-api.ts`、`src/pages/AdminPage.tsx`、`src/pages/ResourcesPage.tsx`、新增若干 `src/components/admin/*Section.tsx`
- **验收标准**：① 后台可增删改首页全部区块内容 + 资源文章；② 前台刷新即生效；③ 含防误删确认。
- **工作量**：XL　**价值**：★★★★★　**状态**：已实现（代码完成 + tsgo/biome/生产构建均通过；待应用建表迁移 SQL 到生产库后即可手测上线，2026-06-14）

- **背景**：前台无连续学习、本周时长等留存钩子；排行榜仅管理员可见。

## 六、关键设计决策（已确认，记录结论）

> 2026-06-13 三项决策已与运营者确认，后续实现以此为据。

1. **决策 A：主页对"已登录老用户"的分屏方式 —— 已采纳推荐**
   HomePage 顶部插入老用户学习首屏（R-P0-01），介绍型内容（俱乐部介绍/创始人/课程体系/定价等）整体下移保留。**不**把 `/` 重定向到 `/learning`——保留首页对新付费用户首次到访的信任建立价值。

2. **决策 B：运营后台能力边界 —— 已确认**
   - **已有（无需重复）**：课程创建/编辑/归档（`CourseManagementSection`，含封面URL、视频URL、分类、等级、时长、学分、会员类型、试看）；视频采用 R2 上传 + 后台填 URL 模式（凭证见 `.env.upload`）；用户会员等级管理（`StudentListSection`）。

3. **决策 C：学习激励的开放度 —— 已确认（隐私优先）**
   排行榜/学习数据**暂不向全体会员开放**（学员/老师有隐私顾虑）。个人可见自己的学习数据；排行榜**仅管理员可见**（已是现状）；激励以个人数据为主（连续学习、本周时长、个人成就/里程碑，均仅自己可见）。
