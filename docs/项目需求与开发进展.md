## 0. 使用规则

以后不再新建“接手文档”“需求池”“本 session 进展”或独立的功能进度 Markdown。所有管理信息统一写在这里：

1. 已实际修改的内容写入“实际修改”区，必须包含日期、变更内容和证据。
2. 尚未实现或只完成一部分的内容写入“计划与待办”区，不得混入已完成表。
3. 一项计划进入代码后，先把已完成切片移入实际修改区；剩余范围继续留在计划区。
4. “代码已合并”只表示已进入 Git 历史，不自动等同于“远端数据库已迁移”或“线上已验收”。
5. 测试报告、技术机制、部署手册、课程大纲和语料可以保留为支撑资料，但不得在其中维护另一套状态或路线图。

### 状态词

| 状态 | 含义 |
|---|---|
| 已落盘 | 本地资料或实现已经完成并可核对，但尚未进入 Git 历史，也不代表已部署。 |
| 已合并 | 功能代码已在 `master` 的提交历史中 |
| 已合并并有线上记录 | 历史记录明确写有迁移、函数部署或线上探针；本轮未重新联网复核 |
| 部分完成 | 已有可用实现，但原需求仍有明确缺口；缺口同时登记在计划区 |
| 待评审 | 只有需求设想，尚未确认进入开发 |
| 已排期 | 已确认要做，但尚未开始 |
| 开发中 | 存在未完成代码，尚未达到验收口径 |
| 暂缓 | 需求有效，但当前不投入开发 |
| 退役 | 与当前产品定位冲突，明确不做 |

## 1. 当前项目基线

### 1.1 产品定位

- 网站主要维护已付费老用户，降低回到课程和持续学习的摩擦。
- 拉新、成交和付款主要发生在外部平台；本站不继续投入以转化为目的的功能。
- 运营者是单人、非技术运营者，后台可配置、可预览、防误操作优先于硬编码页面。
- 学习数据默认个人可见；会员排行榜保持管理员可见。
- HAI 当前以 `/hai` 的“问问哈老师”单聊为核心；圆桌与素材上传后端能力保留，但不作为当前前端主入口。

## 4. 关键产品与架构决策

### 4.1 主页与课程

- 已登录用户优先看到学习首屏；介绍型内容下移，不把 `/` 强制重定向到 `/learning`。
- 首页动态流与个人通知是两套能力：前者负责公开更新，后者负责具体用户触达。
- Plus 主干固定为“理论篇 / 教学设计原理篇 / 场景篇”；Pro 教师 AI 课与 Plus 结构隔离。
- Plus 篇章和模块以数据库为主、前端兜底；课程内容以 `教学通识课Plus-课程大纲.md` 为依据。
- 移动端课程详情使用固定课程控制栏；桌面端保持目录与正文分区滚动。

### 4.2 HAI

- Chat 主链路只执行模块绑定且已发布的 Skill；Skill 不可用时明确报错，不再进入 Context Orchestrator 或旧 Prompt。Mac 本地路径只记录来源，Edge Function 只能执行数据库中的版本快照。
- Chat Skill 的版本边界包含 SKILL.md、reference 文件正文与加载策略；发布生成完整 manifest/hash，运行时只能读取同一 published 版本，不跨版本拼接 reference。
- `ask-han` 的意图识别与方法卡选择由 Chat Skill 运行链完成；不再维护独立的上下文编排/发布 Prompt 回退分支。
- 不知道教材或文章正文时明确说不知道，再转向教学判断和分析框架。
- Skill 调整先复制为草稿、离线验证、逐例保存和基线对比；已发布版本及其 references 保持不可变。
- 线上运行配置可以留在 Supabase；代码、迁移、方法卡源码和前端变更必须进入 Git。

## 6. 接手顺序

1. 先读本文件的“当前项目基线”“实际修改”“计划与待办”。
2. 用当前代码和 Git 状态复核准备修改的功能，不沿用旧 session 的“待发布”判断。
3. 从计划表选一个最小可完成切片；默认优先级为 P0 → P1 → P2。
4. 完成代码、迁移或配置后执行与风险相称的测试。
5. 把完成事实写入第 2 节，把剩余缺口保留在第 3 节，并更新顶部日期。
6. 不再创建新的接手、需求或进展文档。
